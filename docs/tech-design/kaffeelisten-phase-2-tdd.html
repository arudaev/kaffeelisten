1. Summary

Kaffeelisten is a tablet app that replaces the paper coffee-consumption sheet at ITC1 Deggendorf. The first version was built at the hackathon on placeholder data. Phase 2 turns it into a maintained product for real day-to-day use on the campus.

This document describes, for the team and for the people running the campus, exactly what changes and how. There are six pieces of work: (1) replace the demo data with real campus data, (2) require full contact details when adding a person, (3) a 6-digit access code (PIN) that the admin can change and reset themselves, (4) the campus CEO automatically copied on every monthly report, (5) a personal monthly email to each person who consumed something, and (6) a settings screen so all of this can be managed without a developer.

Work already finished on the branch is marked DONE; the rest is PLANNED.

2. Goals & non-goals

Goals

  • Run on real campus data — real companies, people, products and prices.
  • Let non-developers operate it: change/reset the PIN, manage who gets reports, turn features on/off — all in the app, no redeploy.
  • Reach every person who consumed something at month-end with their own statement, without making them log in.
  • Stay on free service tiers, and keep logging a coffee under 15 seconds.

Non-goals (deliberately out of scope)

  • No personal logins or accounts for members — the member email is one-way only.
  • No invoicing or billing automation — the CEO still bills companies; the app only provides the figures.
  • No multi-location support — ITC1 only.
  • No per-person admin accounts this phase — one shared admin PIN (see §13).

3. Current system

In plain terms: a tablet app, a database in the cloud, and a small set of automated jobs that send the monthly report.

The app is a web app (React) running in the browser/on a wall-mounted tablet. Data lives in Supabase (a hosted PostgreSQL database). Small server-side functions on Vercel handle anything sensitive — building and emailing the monthly report (PDF + Excel via Resend), then archiving the month and clearing it. A scheduled job runs on the last day of the month as a safety net in case the admin forgets. The admin PIN and the report recipient are currently fixed values set by a developer.

What's wrong with that today

  • The PIN and recipients can only be changed by a developer doing a redeploy.
  • The PIN is 4 digits, fixed, stored in plain text, with no way to reset it.
  • A person's email is optional, so we can't reliably reach individuals.
  • The settings screen is empty — a placeholder.

4. Data model changes

New app_settings table DONE · migration 010

A single row that holds all admin configuration — moving it out of developer-only settings and into the database so it can be managed from the app. Because it holds secrets (the access code), it is locked down so the browser can never read it; only the trusted server-side functions can.

-- migration 010_app_settings.sql (abridged)
create table app_settings (
  id                        smallint primary key default 1 check (id = 1),
  admin_pin_hash            text,                -- access code, stored hashed (never plain)
  pin_length                smallint not null default 6,
  pin_updated_at            timestamptz,
  pin_reset_token_hash      text,                -- one-time reset code, hashed
  pin_reset_expires_at      timestamptz,
  report_recipients         text[]  not null default '{}',   -- who gets the report
  ceo_email                 text,
  cc_ceo_on_reports         boolean not null default true, -- CEO copied on every report
  member_statements_enabled boolean not null default true, -- personal emails on/off
  updated_at                timestamptz not null default now()
);
alter table app_settings enable row level security;  -- browser fully denied
grant select, insert, update on app_settings to service_role;

Person's email becomes required DONE · migration 011

Personal monthly emails only work if every person has an email on file. Migration 011 makes the email mandatory in the database. It checks first and stops with a clear message if any person is still missing one, so it is applied after the demo data is cleared or all emails are filled in (see §10).

5. PIN management PLANNED

The 6-digit code that opens the admin area — changeable and resettable by the admin, with no developer involved.

How it's stored

The PIN is stored as a one-way hash (it cannot be read back, only checked), and is never sent to the browser. The old fixed code keeps working only until the admin sets a real PIN in the app for the first time.

Server functions

FunctionPurpose
verify-pinCheck an entered PIN against the stored hash (falls back to the old fixed code until one is set).
change-pinSet a new 6-digit PIN after confirming the current one.
request-pin-resetEmail a one-time reset code (valid 15 minutes) to the admin/CEO addresses.
reset-pinSet a new PIN using that reset code — or a private recovery code as a last resort.

Reset — two ways (decided)

Locked outAdmin forgot the PIN, taps "PIN zurücksetzen".
→
Email codeA one-time code is emailed, valid for 15 minutes.
→
New PINCode entered, a new 6-digit PIN is set.
→
BackstopIf email is unavailable, a private recovery code resets it.

6. Recipients & CEO CC PLANNED

The monthly report will be sent to the configured recipient list, with the CEO automatically in copy (CC) whenever that option is on. Because both the manual "send report" button and the automatic month-end job use the same routine, the CEO is copied on every report by design — manual or automatic.

7. Per-member monthly statements PLANNED

This adds to the existing report; it does not replace it. The company report to the admin and CEO stays exactly as is. In addition, each person who logged at least one item during the month receives their own itemized statement by email at month-end. This can be switched off with a single setting.

1 · Company reportSent to admin + CEO as today.
→
2 · GroupThe month's entries grouped per person.
→
3 · Personal emailOne itemized email per person (their entries only).
→
4 · ArchiveMonth archived and cleared as today.
  • Volume: about one email per active person per month — comfortably within the free email allowance even at the full campus roster.
  • Reliability: if one address fails, it is logged and skipped; the rest still go out, and the company report has already been sent.
  • Privacy: a person sees only their own consumption — never anyone else's, and never the company total.

8. Settings page & API PLANNED

The empty Settings tab becomes a real screen, backed by a settings function that never returns the stored PIN or reset code. From there the admin can:

  • Change the PIN, or start a reset.
  • Edit who receives the report, set the CEO email, and toggle the CEO copy.
  • Turn the personal member emails on or off.
  • See read-only status: when the last report was sent, and the current PIN length.

9. Security

  • Secrets stay server-side: the settings table is locked so the browser can never read the PIN; it is never included in any data sent to the app.
  • Hashing: the PIN and the reset code are stored hashed, and compared safely.
  • Guarding against guessing: a 6-digit code has a limited number of combinations, so the check is rate-limited (it slows down / locks out after repeated wrong attempts). This is the main new risk versus the old fixed code and ships together with the PIN feature.
  • Reset codes: single-use, expire after 15 minutes, stored hashed, cleared once used.
  • Recovery code: a private last-resort code kept outside the database; rotated if ever exposed.
  • PINs and codes are never written to logs.

10. Migration & rollout

The order matters — the destructive steps are done by hand, once, when the real data is ready.

  1. Apply migration 010 (new settings table) — safe anytime; the app keeps working.
  2. Export anything worth keeping from the archive (to a spreadsheet).
  3. Run the data-reset script — clears the demo data; settings are kept.
  4. Load the real data — companies, people (with emails), products and real prices.
  5. Apply migration 011 — locks in the "email required" rule once no one is missing one.
  6. Set the admin PIN from the app once the PIN feature ships.
Steps 3–5 are destructive and done manually — never automatically by a deploy. The reset has not been run against the live system yet.

11. Failure modes & edge cases

SituationWhat happens
A person has no email (old record)Their statement is skipped and logged; migration 011 prevents new such records.
One member email address is rejectedLogged and skipped; everyone else still receives theirs.
A month's report is sent twiceArchiving ignores duplicates; personal emails would re-send (see §13).
No PIN set yet (fresh install)The old fixed code is accepted until a PIN is set.
Reset code expired or already usedRejected; request a new one or use the recovery code.
Scheduled job runs but it's not month-endIt does nothing and exits (existing behaviour).

12. Alternatives considered

  • Keep the PIN as a fixed developer setting: rejected — it can't be changed or reset by the people running the campus, which is the whole point.
  • Full login system for admins: heavier than needed for a single shared admin role; revisit only if per-person admin accounts are wanted (§13).
  • Personal emails replace the company report: rejected — the CEO still needs the per-company view for billing. We keep both.
  • Personal email with total only (no item list): a simpler option that's still on the table (§13).

13. Decisions & open points

Decisions already taken on this branch:

  • Personal member emails add to the company report rather than replacing it.
  • PIN reset offers both an emailed code and a private recovery-code backstop.
  • The PIN is 6 digits and all member fields (name, email, company) are required.

Points still worth a quick confirmation from the campus side:

  • Statement detail: full itemized list (current plan) or just the monthly total?
  • Admin access: is one shared PIN enough, or are separate logins per person needed later?
  • Recipients: which address receives the report, and which CEO address is copied?
  • Re-sends: do we need a safeguard so personal emails can't go out twice if a report is re-run?
  • Data & privacy: are personal emails acceptable, and are there retention expectations?
